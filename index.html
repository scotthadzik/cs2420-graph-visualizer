      Dijkstra’s — Shortest Path Tree

      Source: Noorda (A) · minimizes distance from the source to every other vertex

      Tree edges: A–B(4), A–C(2), C–E(1), B–D(1)
      Sum of edge weights: 8

      Prim’s — Minimum Spanning Tree

      Start: Noorda (A) · minimizes total edge weight across the whole tree

      MST edges: A–C(2), B–C(3), C–E(1), B–D(1)
      Total weight: 7
      Key difference: Both trees connect all 5 buildings and share 3 edges (A–C, C–E, B–D). They disagree on how to reach Engineering Tech (B):
      • Dijkstra’s uses A–B(4) — the direct edge — because that IS the shortest path from Noorda to Engineering Tech (the detour A → C → B would cost 2 + 3 = 5, worse).
      • Prim’s uses B–C(3) instead. It doesn’t care about paths from the source — it just wants the cheapest edge that connects a new vertex. Swapping 4 for 3 saves 1 unit of total weight.
      The trade-off: Prim’s MST is lighter (7 vs 8), but the path from Noorda to Engineering Tech is now longer (2 + 3 = 5 instead of 4). Dijkstra’s pays more total weight to guarantee every single path from the source is optimal.

      🏫 Real-World: Navigating Weber State Campus

      Vertices = buildings  ·  Edges = walking paths  ·  Weights = walking minutes

      Dijkstra’s — “Get me there fast”

      “I’m at my dorm. Chem class starts in Lind Lecture Hall at 9am. What’s the fastest route?”

      → Run dijkstra("dorm"). Read off dist["Lind"] — that’s your walking time, and the tree gives you the path.

      Prim’s — “Connect everything cheaply”

      “Facilities is running fiber-optic between every building on campus. What’s the minimum cable length?”

      → Run prims_mst("any_building"). The MST edges are the cable runs; the total weight is the cable you’ll buy.

      ☕ The Starbucks twist: “Same trip, but I want to grab coffee at the Tracy Hall Starbucks on the way to chem class.”

      → Still Dijkstra’s — but run it twice: dist["Tracy"] from dorm, then dist["Lind"] from Tracy Hall, and sum them. Prim’s MST can’t answer this — the MST edges aren’t necessarily the shortest routes between any two vertices.

      Mental model:
      • “Fastest X → Y” = Dijkstra’s (one run)
      • “Fastest X → stopover → Y” = Dijkstra’s (two runs, summed)
      • “Connect all of these cheaply” = Prim’s
      Adjacency list — the underlying data structure

      This is the unordered_map<string, vector<Edge>> the algorithm actually queries. It doesn’t change during the run — both Dijkstra’s and Prim’s just read from it when they ask for a vertex’s neighbors.
